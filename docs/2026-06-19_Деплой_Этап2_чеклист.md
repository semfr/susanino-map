# Карта Сусанино — чеклист готовности к деплою (Этап 2)

Дата: 2026-06-19
Статус: **ПОДГОТОВКА к деплою** (не сам деплой). Документ сводит две независимые аудиторские проверки (веб-раздача и бот/эксплуатация), убирает дубли и упорядочивает по приоритету.

Архитектура (контекст):
- **web** — Next.js 16.2.1 с `output: 'export'` → статика в `web/out`, раздаётся nginx (Node-сервер/`next start` не нужен).
- **Лубок** — через `ImageOverlay` (тайлы убраны, `public/tiles/lubok` — мёртвый груз).
- **bot** — Python aiogram 3.26, читает данные из корневого `data/` (синхронизируется из `web/src/data` скриптом `tools/sync_data.py`).
- Токен бота сейчас лежит в `bot/bot_api.txt`; `.env` пока нет.

Легенда серьёзности: 🔴 high (блокер деплоя) · 🟡 med · 🟢 low.

## А. Что уже готово

- **Сборка web в статику** работает: `web/out/` содержит `index.html`, `catalog`, `object`, `_next`, `404.html`, `sw.js`, `manifest.json`, `favicon.ico`, `icons/`.
- **PWA-каркас на месте**: `manifest.json` с базовыми полями (`name`, `short_name`, `display: standalone`, цвета) и иконками 192/512 (`web/public/icons/icon-192.png`, `icon-512.png`); зарегистрирован `sw.js`.
- **nginx базово настроен**: SPA-fallback (`try_files`), gzip, кэш картинок (30d) и JS/CSS (7d), security-заголовки (`X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`). `_next/static` отдаётся через `try_files`. Проксирования на `:3000` нет — и не нужно (статика).
- **systemd-unit `susanino-bot.service` валиден** по синтаксису: `Type=simple`, `User=www-data`, `Restart=always`, `EnvironmentFile=/opt/susanino-bot/.env`. (Не взлетит, пока не закрыты зависимые пункты — см. раздел Б.)
- **Зависимости бота достаточны по нижней границе**: `aiogram>=3.4`, `python-dotenv>=1.0` (есть в `bot/requirements.txt`). Dev-зависимости вынесены в `requirements-dev.txt` (`pytest`, `pytest-cov`).
- **Гигиена git**: корневой `.gitignore` исключает `node_modules/`, `.next/`, `out/`, `.env`, `__pycache__/`, `*.pyc`, `venv/` и **`bot_api.txt`** (токен из git исключён). Проблема в том, что rsync `.gitignore` НЕ учитывает — см. Б-4.
- **config.py читает `.env`** через `python-dotenv` (`load_dotenv(... / ".env")`) и берёт `BOT_TOKEN` из окружения — механизм правильный, не хватает самого файла `.env` на проде.

## Б. Исправить в коде/конфигах ДО деплоя

### 🔴 Б-1. Путь к данным у бота не совпадает с тем, куда их кладёт deploy.sh
`bot/config.py:11` → `DATA_DIR = Path(__file__).parent.parent / "data"`. На VPS бот лежит в `/opt/susanino-bot/`, значит `parent.parent` = `/opt`, и `DATA_DIR` резолвится в **`/opt/data`**. А `deploy.sh:8` rsync-ит данные в **`/opt/susanino-data/`**. Каталога `/opt/data` на сервере нет.
`bot/services/data_loader.py:19-21` читает `config.json`/`categories.json`/`objects.json` **при импорте модуля** → бот падает с `FileNotFoundError` на старте (и попадает в краш-луп из-за `Restart=always`).
**Фикс:** сделать `DATA_DIR` настраиваемым через env (`DATA_DIR = Path(os.environ.get("DATA_DIR", Path(__file__).parent.parent / "data"))`), в `.env` на проде задать `DATA_DIR=/opt/susanino-data`. Альтернатива — rsync данные в `/opt/data` (или в `bot/../data`), но env-вариант чище.

### 🔴 Б-2. На проде нет `.env` с `BOT_TOKEN` — сервис не поднимется
Unit задаёт `EnvironmentFile=/opt/susanino-bot/.env`, `config.py:8` делает `os.environ["BOT_TOKEN"]` (это `KeyError` при отсутствии). Файлов `.env`/`.env.example` нет, `deploy.sh` их не создаёт. Токен лежит только в `bot/bot_api.txt`, который `config.py` НЕ читает.
**Фикс:**
- создать на сервере `/opt/susanino-bot/.env` с `BOT_TOKEN=...` (и `DATA_DIR=/opt/susanino-data` из Б-1), права `600`, владелец `www-data`, вне git и вне rsync;
- добавить в репозиторий `bot/.env.example` (без секретов) как шаблон.

### 🔴 Б-3. deploy.sh не запускает `tools/sync_data.py` перед выгрузкой данных
`deploy.sh` собирает web и rsync-ит `data/` напрямую, но `python tools/sync_data.py` НЕ вызывается. Без ручной пересинхронизации на прод уедут устаревшие JSON и `illustrations/`.
**Фикс:** добавить `python tools/sync_data.py` **до** `rsync data/` (и до сборки web, чтобы синк попал и в `web/out`, если данные используются на фронте).

### 🔴 Б-4. deploy.sh утекает токен и мусор через `rsync bot/`
`deploy.sh:9` → `rsync -avz bot/ user@vps:/opt/susanino-bot/` копирует **всю** папку, включая `bot_api.txt` (токен!), `__pycache__/`, `.pytest_cache/`, `tests/`, `requirements-dev.txt`, `pytest.ini`. rsync `.gitignore` не учитывает. Флага `--delete` для `bot/` нет (старые файлы остаются на проде).
**Фикс:** добавить `--exclude` для `bot_api.txt`, `__pycache__/`, `.pytest_cache/`, `tests/`, `*.pyc`, `requirements-dev.txt`, `pytest.ini`, `.env`. Токен — только через `.env` (Б-2). Рассмотреть `--delete` для `bot/` и `data/` (с осторожностью, чтобы не снести `.env` и `venv/` — их и так нет в источнике, но `--delete` удалит на проде всё, чего нет в источнике; исключить `.env` и `venv/` из удаления через `--exclude`).

### 🔴 Б-5. deploy.sh не создаёт venv и не ставит зависимости
Unit запускает `/opt/susanino-bot/venv/bin/python bot.py`, но `deploy.sh` нигде не создаёт `venv` и не ставит `requirements.txt`. При первом деплое интерпретатора по пути не будет → `ExecStart` не стартует.
**Фикс:** добавить (идемпотентно, для первого деплоя) создание venv и установку зависимостей на сервере, например:
`ssh user@vps "cd /opt/susanino-bot && python3 -m venv venv && venv/bin/pip install -r requirements.txt"` перед `systemctl restart`.

### 🔴 Б-6. sw.js: захардкоженный кэш `v1`, cache-first, нет очистки — отдаёт старую версию после деплоя
`web/public/sw.js:1` → `CACHE_NAME = 'susanino-map-v1'` (захардкожен). Стратегия cache-first (`caches.match(...) || fetch(...)`), нет обработчика `activate` с очисткой старых кэшей. После деплоя пользователи получают устаревшие HTML/`_next`-ассеты, пока вручную не сбросят кэш.
**Фикс:** имя кэша с билд-хэшем/версией (меняется на каждый деплой), обработчик `activate` с удалением старых кэшей, **network-first для HTML и `_next`** (статику с хэшами в имени можно cache-first). См. также раздел В (версионирование SW).

### 🟡 Б-7. objects.json ссылается на отсутствующие картинки (9 × 404)
`data/objects.json` ссылается на `/images/objects/<slug>/main.webp` для 9 объектов (`chasovnya`, `church-voskreseniya`, `dom-osipova`, `domnino-church`, `isupovskoe-boloto`, `konezavod-medvedki`, `ohotino`, `petrovskaya-igrushka`, `susanino-museum`). На диске каталога `web/public/images/objects/` НЕТ — есть только `web/public/images/map/`. Все 9 ссылок дадут 404 и в вебе, и (через rich-сообщения) в боте.
**Фикс:** добавить реальные `main.webp` по этим путям, либо положить общий `placeholder.webp` и временно указывать на него, либо убрать ссылки до появления фото.

### 🔴/🟡 Б-8. nginx: нет HTTPS/443/certbot/редиректа (см. также раздел В)
`deploy/nginx.conf` слушает только `listen 80`. Нет блока `443 ssl`, путей к сертификатам certbot, редиректа 80→443. По HTTP **service worker и установка PWA не активируются** (требуют HTTPS). Это блокер для PWA-функциональности, хотя сам сайт по HTTP откроется.
**Фикс:** выпустить сертификат (`certbot --nginx`), добавить блок `listen 443 ssl` с путями к сертификату, редирект 80→443. Подробности по renew/таймеру — в разделе В.

### 🟢 Б-9. nginx: нет long-cache для `_next/static`
`_next/static` отдаётся `try_files`, но попадает под общее правило JS/CSS (`7d`). Ассеты Next имеют хэш в имени и безопасны для `immutable` на год.
**Фикс (опционально):** `location /_next/static/ { expires 1y; add_header Cache-Control "public, immutable"; }`.

### 🟢 Б-10. manifest.json: нет `id`, `scope`, `maskable`
Базовые поля и иконки 192/512 есть, но нет `id`, `scope` и `maskable`-иконки. Установка PWA всё равно требует HTTPS (Б-8).
**Фикс:** добавить `"id": "/"`, `"scope": "/"` и хотя бы одну иконку с `"purpose": "maskable"`.

### 🟢 Б-11. Leaflet CSS грузится с CDN (unpkg)
`layout.tsx` подключает `leaflet.css` с unpkg → в офлайне (PWA) теряются стили карты.
**Фикс:** локализовать `leaflet.css` (положить в `public`/импортировать локально), убрать внешний CDN.

### 🟢 Б-12. requirements.txt не запинен
`aiogram>=3.4`, `python-dotenv>=1.0` — нижняя граница ок, но версии открытые → риск, что на сервере встанет несовместимая мажорная версия.
**Фикс:** запинить точные версии (`aiogram==3.26.x`, `python-dotenv==1.0.x`). Dev-зависимости держать только в `requirements-dev.txt` (и не выгружать на прод — см. Б-4).

### 🟢 Б-13. Мёртвые тайлы лубка
Лубок теперь через `ImageOverlay`, каталог `web/public/tiles/lubok` лишний (раздувает `web/out` и rsync).
**Фикс (опционально):** удалить `web/public/tiles/lubok`. Удаление файлов — только с явного подтверждения пользователя.

## В. Эксплуатация (H-006): HTTPS, мониторинг, версии SW, runbook

### 🔴 В-1. Выпуск сертификата и автопродление
- Выпустить сертификат: `certbot --nginx` для `susanino-map.ru` и `www.susanino-map.ru` (см. Б-8).
- Проверить автопродление: `systemctl status certbot.timer` (должен быть активен), тест — `certbot renew --dry-run`.
- В nginx добавить блок 443 + редирект 80→443.

### 🟡 В-2. Мониторинг аптайма и алертинг
Сейчас нет healthcheck сайта, проверки живости бота и алерта. При перманентной ошибке `Restart=always` даёт **краш-луп без уведомления**.
- Пинг сайта (HTTP 200 на `/`) и проверка процесса/логов бота.
- Алерт в Telegram при недоступности (можно отдельным чатом/ботом).

### 🔴 В-3. Версионирование service worker
См. Б-6: имя кэша с билд-хэшем + `activate`-очистка + network-first для HTML/`_next`. Это эксплуатационный must — без него каждый деплой «застревает» у пользователей.

### 🟡 В-4. Runbook восстановления — `deploy/RUNBOOK.md`
Нет инструкций на инциденты. Написать `deploy/RUNBOOK.md` с командами:
- перевыпуск сертификата (`certbot renew` / `--force-renewal`);
- восстановление `.env` (шаблон, права 600, владелец www-data);
- диагностика бота (`journalctl -u susanino-bot -n 100`, проверка `DATA_DIR`, импорт `data_loader`);
- восстановление данных (повторный `sync_data.py` + rsync);
- откат (предыдущая версия `web/out`, рестарт сервисов).

## Д. Что должен предоставить ПОЛЬЗОВАТЕЛЬ (решения и вводные)

Без этих данных деплой Этапа 2 невозможно ни начать, ни автоматизировать. Пункты со звёздочкой (*) блокируют сразу несколько технических шагов.

1. **Решение, где хостить*** — конкретный VPS-провайдер и тариф. От этого зависит всё ниже.
2. **VPS и его ОС*** — IP-адрес/хостнейм сервера, дистрибутив и версия ОС (ожидается Linux: Ubuntu/Debian — для certbot/systemd/rsync), SSH-доступ (пользователь и способ авторизации). Сейчас в `deploy.sh` стоят плейсхолдеры `user@vps` — нужны реальные значения.
3. **Домен*** — подтверждённое доменное имя. В nginx.conf зашиты `susanino-map.ru` и `www.susanino-map.ru` — подтвердить, что домен именно этот и он зарегистрирован/управляется пользователем.
4. **DNS A-записи** — настроить (или дать доступ настроить) A-записи `susanino-map.ru` и `www.susanino-map.ru` на IP VPS. Без корректного DNS certbot не выпустит сертификат.
5. **Прод `BOT_TOKEN`** — продакшен-токен Telegram-бота (для `/opt/susanino-bot/.env`). Решить: использовать текущий токен из `bot_api.txt` или завести отдельного прод-бота. Токен в чате/документах НЕ печатать.
6. **E-mail для certbot/Let's Encrypt** — адрес для регистрации сертификата и уведомлений об истечении.
7. **Telegram-канал для алертов (по В-2)** — chat_id/бот для уведомлений о падении сайта/бота (если согласуем мониторинг на Этапе 2).
8. **Фото объектов (по Б-7)** — решение: предоставить 9 реальных `main.webp` или согласовать временный placeholder.
